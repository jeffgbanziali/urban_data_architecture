Vue d'ensemble de l'architecture

  • Data Lake MinIO : zones bronze (brut, horodaté), silver (nettoyé, Parquet), gold (agrégé, dont un GeoJSON enrichi).
  • PostgreSQL : base Gold (indicateurs, biens, utilisateurs, agrégats temps réel) avec contraintes d'intégrité et rôle en lecture seule pour l'API.
  • Kafka : deux topics — transactions.raw (transactions immobilières) et events.stream (qualité de l'air quasi temps réel).
  • Géocodage réel : adresses d'espaces verts géocodées via l'API BAN, puis rattachées à leur arrondissement par un vrai test point-in-polygon (Shapely) sur les géométries officielles OpenData Paris.
  • Push temps réel sans polling : le consumer Kafka notifie PostgreSQL (pg_notify) après chaque événement ; l'API écoute (LISTEN via asyncpg) et relaie instantanément aux clients WebSocket.
  • Airflow : orchestration des DAGs ingestion_bronze → transform_silver → aggregate_gold.
  • FastAPI : API REST sécurisée (JWT, rôles client/employé/admin) documentée sur /docs, incluant GET /geo/arrondissements qui ferme la boucle vers le frontend.
  • Frontend React : vitrine agence (biens, favoris), espaces par rôle, et un explorateur de données branché sur l'API (carte choroplèthe sur géométries officielles, 6 indicateurs adossés à de vraies sources, sélecteur d'année pour le prix).

Méthodologie de préparation des données (C3.1)

  1. Typage explicite des colonnes (dates, numériques, catégorielles).
  2. Suppression des doublons sur la clé métier.
  3. Imputation des valeurs manquantes par la médiane du groupe (arrondissement).
  4. Filtrage des valeurs aberrantes (prix/m² hors de [1 000 € ; 30 000 €]).
  5. Géocodage des adresses (espaces verts) via l'API BAN, puis résolution de l'arrondissement par point-in-polygon sur la géométrie officielle (pas une approximation par centroïde).
  6. Validation de schéma avant promotion en zone Silver.
  7. Rapport de qualité JSON versionné à chaque exécution (s3://silver/_reports/quality/), incluant le taux de réussite du géocodage.

Limites et améliorations possibles

  • L'explorateur de données est branché sur l'API (GET /geo/arrondissements) ; reste une évolution possible : positionner individuellement les espaces verts géocodés sur la carte (seul leur comptage par arrondissement est exploité pour le moment).
  • delits_pour_1000_hab : vraie source SSMSI disponible sur data.gouv.fr (fichier national ~38 Mo, schéma complexe) — à brancher dans une itération future.
  • score_transport : l'API RATP Open Data référencée a été dépréciée (dataset supprimé) — à rebrancher si une nouvelle URL officielle est publiée.
  • part_logements_sociaux_pct, revenu_median_annuel : disponibles via les recensements INSEE (RP), mais pas encore intégrés dans le pipeline.
  • Les sources espaces verts (OpenData Paris) et DVF (geo-dvf, data.gouv.fr) utilisent leur vrai format en priorité, avec repli automatique sur des données synthétiques DVF si le téléchargement échoue.
  • Le push WebSocket ouvre une connexion PostgreSQL dédiée par client (LISTEN) ; un usage à très forte concurrence préfèrerait un listener interne unique avec fan-out.
  • Authentification par clé secrète unique côté JWT (pas de rotation de clé ni de refresh token) — suffisant pour la démonstration.
  • Limitation de débit de l'API en mémoire, non partagée entre plusieurs instances.
